# Sook's Saga Scroll — Feature Ideation

**Date:** 2026-07-19 · **Subject:** new features / improvements for the DDO saga tracker · **Mode:** repo-grounded · **Format:** markdown

Generated by a 3-lens ideation fleet (core tracking workflow · live-data layer · public-launch sharing/onboarding/safety) → 24 candidates → critiqued → **7 survivors**. Web research skipped (niche personal app).

## Grounding context

- **The app:** single-file, offline HTML tracker (no backend/build; localStorage schema v14). Multi-character, per-character themes; Heroic/Epic/Legendary + Non-Saga containers; First-Time-Reaper (permanent) tracking; banked/incomplete rewards; completed rollup (now a fractional progress bar); quest details (story/monsters/traps/puzzles-with-overrides/quest-givers from DDO Wiki); **live DDO Audit integration** (LFM index tiered by reaper skulls, server population, guildmates-online + their current quest, quest autocomplete on in-game entry); audio alerts; filters/search; JSON import/export.
- **Timely context:** just published to a **public GitHub Pages URL** → audience shifts from "just Sook" to any DDO player; each visitor gets their own empty per-origin localStorage; there is **no account system and cannot be one** (no backend).
- **Known data gap:** many quests still show "Not yet sourced" wiki details.
- **Topic axes:** core tracking workflow · live-data / real-time layer · data safety & portability · onboarding & sharing · delight.
- **Hard constraints every idea respects:** stays single-file/offline; no backend/server; only live source is the public DDO Audit API; data is per-browser localStorage.

---

## Ranked survivors

### 1. Data-safety pack for the public origin switch  *(axis: data safety — highest timeliness)*
**What:** Three small guards that matter the moment real visitors arrive: (a) an auto-backup **nag** ("your data is only as safe as your last export") with one-tap dated JSON download; (b) a **trust + storage-health** line ("100% offline — stored only in this browser") plus an incognito/quota warning; (c) an **import guardrail** that checks schema version and confirms "this replaces/merges your N characters" before overwriting.
**Basis:** Public visitors on the new origin start empty and won't grasp that localStorage is the *only* copy — a cache clear, incognito tab, or new device silently erases everything, and a naive import could clobber a stranger's whole scroll. Directly protects the users the launch exists to serve.
**Effort:** S–M. All client-side (`navigator.storage.estimate()`, existing JSON import/export, the existing v14 versioning). Fits the constraints cleanly.
**Bold stretch (same theme):** **"Cartridge Save"** — export a *standalone HTML copy with the data baked in* (TiddlyWiki / emulator save-state model), so the file itself is the account: reopen your full tracker on any device, offline, even if the site's down. Effort L, but the single-file architecture makes it uniquely natural here.

### 2. Next-best-action engine (saga-overlap + forecast)  *(axis: core workflow — flagship value)*
**What:** Turn the quest→saga graph the app already holds into guidance: a per-character **"run this next"** (the quest that advances the most incomplete sagas / is closest to banking), a per-saga **"distance to reward"** readout (N quests + M FTRs remaining, shortest path highlighted), and a browsable **overlap map** (quests ranked by how many of your incomplete sagas they touch).
**Basis:** DDO saga overlap is the core optimization most players never fully internalize; the app has the full graph but currently makes you eyeball the whole scroll to decide "what do I run tonight." Pure client-side derivation of data already in localStorage.
**Effort:** M. Offline-clean; the app's unique data asset finally made actionable.

### 3. Attention strip: almost-done sagas + unclaimed FTRs  *(axis: core workflow — best ROI quick win)*
**What:** A pinned top-of-scroll strip surfacing sagas one or two quests from banking, and quests you've **already completed on reaper but never claimed the First-Time-Reaper** — free reaper points left on the table.
**Basis:** A concrete recurring pain: leaving a saga 95% done and never banking it, or forgetting an unclaimed FTR. "You already did the hard part — go collect." Pure filter over existing state.
**Effort:** S. Highest value-to-effort of the set.

### 4. Quest-first opportunistic view (tied to live LFMs)  *(axis: live-data — non-obvious inversion)*
**What:** Invert the char→saga layout: given a live forming group (or any quest), instantly show **which of your characters benefits most** — which sagas it advances and whether FTR is still unclaimed for each — and a **reaper "shopping list"** that surfaces only LFMs satisfying an outstanding FTR need, ranked by how many goals each knocks out. Plus a **guildmate "catch them before they finish"** ping when a guildmate starts a quest you still need.
**Basis:** The real-time decision is quest-first ("an R4 Shroud LFM just popped — who should I log in?"), but the app is organized character-first. Fuses the existing FTR tracker with the reaper-tiered LFM index and guildmate feed — no new data source.
**Effort:** M. Re-slices feeds already polled; offline-degradable.

### 5. Public onboarding: demo character + coach overlay  *(axis: onboarding)*
**What:** On empty storage only, seed a clearly-labeled read-only **"Sook the Sample"** character mid-progress (with a one-click "clear demo & start mine"), plus a dismissible 3–4-step wax-seal-styled coach (name a character → pick a theme → add a saga → optionally connect live data). Include **bulk quick-entry** (mark a whole saga complete; copy a completion baseline from another character) to kill the biggest adoption friction: initial data entry.
**Basis:** Every public visitor lands on a blank parchment with no idea what "good" looks like; veterans dread clicking through dozens of already-done quests. No backend = the app itself must teach and speed entry.
**Effort:** M. Demo is a hardcoded JSON constant; overlay is DOM + an "onboarded" flag; bulk entry wraps existing toggles.

### 6. Server opportunity-steering  *(axis: live-data — cheap insight)*
**What:** Instead of raw server population, compute a per-server **"opportunity score"** = population **weighted by how many LFMs exist for your tracked quests**, and recommend which server to play on tonight. A high-pop server with zero relevant groups scores below a medium-pop server with three.
**Basis:** Combines two feeds the app already fetches (population + LFM index) into the decision players actually make at login. Low effort, genuinely useful.
**Effort:** S–M. Client-side scoring; offline-degradable.

### 7. "Seal the Saga" ceremony + shareable achievement card  *(axis: delight + growth)*
**What:** On a saga hitting 100%, a short wax-seal-stamping animation (+ existing audio) and a downloadable, theme-colored **achievement card PNG** (rendered client-side via canvas) captioned with character + saga — ready to drop in Discord/guild chat. Optionally a read-only **URL-encoded progress snapshot** (`#fragment` + QR) for bragging/recruiting without exposing all characters.
**Basis:** Turns a private milestone into an on-brand, shareable artifact that pulls other DDO players to the public URL — the cheapest growth loop a no-backend app can have. Leans on existing themes + audio; canvas-to-PNG and URL fragments never touch a server.
**Effort:** M.

## Quick wins worth doing regardless (low effort, folded above)
- **Difficulty-drop / threshold LFM alert** — per-quest "only alert me at ≤ R-N" filter on the existing alert system (kills reaper-elitist noise for capable-but-cautious players). S.
- **Bulk "mark all in saga complete" / copy-from-character** — biggest data-entry friction reducer (folded into #5). S.

## Rejected (with reasons)
- **Best-time-to-login heatmap** — premise (continuously logging LFM sightings over days) doesn't match how an occasionally-opened tab is used; the histogram would be too sparse to trust.
- **"Be the beacon" recruiter** (detect idle players in public zones to whisper) — depends on the DDO Audit API exposing arbitrary players' idle-in-zone locations, which is uncertain; L effort for a payoff that honestly stops at "here's who to whisper."
- **Session/route planner (drag-queue builder)** — overlaps #2 but much heavier; the recommend + forecast delivers most of its value at far lower cost.
- **Progress-history timeline** — pleasant momentum feature but adds schema growth for nice-to-have value; #3 satisfies the same "surface what matters" instinct more cheaply. (Reasonable later follow-on.)
- **Passive ticker / desktop notifications** — modest delta over the existing audio alerts; the useful slice (threshold alerts) is captured as a quick win.
- **LFM fill-velocity watch** — genuinely handy but depends on reliable poll cadence + LFM timestamps; folded as a possible refinement of #4 rather than a standalone feature.

## Recommendation → next step (`/ce-brainstorm`)
Two are worth taking into `ce-brainstorm` first, for different reasons:
- **#1 Data-safety pack** — most *timely*: it protects real visitors the instant the public URL goes live, and it's low effort. The responsible thing to ship alongside the launch.
- **#2 Next-best-action engine** — most *distinctive value*: it turns the app's unique quest→saga graph into guidance nothing else offers.

Recommended: brainstorm **#1** first (right-sized, launch-critical), then **#2** as the flagship follow-up. Ideation identifies directions; `ce-brainstorm` defines the chosen one precisely enough to plan.
